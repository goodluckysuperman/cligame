# <font style="color:#DF2A3F;background-color:#FBDE28;">版本一</font>
## Act 1：故障
凌晨 03:17。  
封闭设施 Chamber 0 报警。  
你接手终端，必须在 04:00 前恢复封印。

系统很正常，甚至有点官僚、冷漠。

## Act 2：帮手
内网有人联系你，说自己是上一班遗留的远程支援。  
他指导你恢复第一个模块。

你照做后，第一次听见那句：

祂是初，祂是终。

这时还像噪声。

## Act 3：裂缝
你在 ARCHIVE 里找到日志。  
里面的操作记录，和你当前正在做的步骤一模一样。  
甚至连“你刚刚打错的命令”都已经写在旧日志里。

这时那句升级成：

祂是初，祂也是归处。

开始有“循环 / 回来过 / 你不是第一次”的味道。

## Act 4：诱导
远程操作员让你恢复 `AUTH`。  
他说这样才能拿到最高权限关门。

你一恢复，系统把你的名字写进配置里。  
终端第一次直呼你：

祂是初，祂是终，而你是门。

这一下不再是诗，而是**系统状态说明**。

## Act 5：反转
你发现所谓“封印系统”，不是用来关住祂的。  
是用来**筛选谁适合作为门**的。  
恢复模块不是修封印，而是在做匹配测试。

而那个远程“帮手”，其实一直在把你往最适合的路径上推。

## Act 6：抉择
04:00 将至。  
你有三个方向：

+  彻底断电：你可能活，但整座设施一起失控 
+  删掉自己的认证：祂进不来，但你会被系统抹成“未登记” 
+  完成最后一步：你成为门，但阻止更坏的东西穿过来 

这就比“读经文被吓”有意思太多，因为这里有：

+  目标 
+  悬念 
+  操作 
+  决策 
+  代价 

# <font style="color:#DF2A3F;background-color:#FBDE28;">版本二</font>
## 一、先把产品定位钉死
### 项目名
**《门限协议》**

英文名可以是：

**Threshold Protocol**

这个名字很合适，因为它同时有三层意思：

+ 协议
+ 阈值
+ 门限 / 门槛 / 门

而你的主题又正好是“你是门”。

---

## 二、核心设计原则
这个游戏不是普通文字冒险，它更像：

> **一款伪终端事故处理游戏**  
玩家需要在有限时间内，使用终端工具探索、判断、执行协议，阻止零号收容室异常扩散。
>

真正的吸引力来自四件事：

+ 有**明面上的任务**
+ 有**可操作的工具**
+ 有**时间压力**
+ 有**信息不可信**

所以最终体验不是“看故事”，而是：

> **我正在处理一场事故，但越处理越发现整个系统都在利用我。**
>

你这个想法非常适合落地。

========================================

[1] 中文  
[2] English  

> 

选择中文后，整个界面都中文化。

### 2. 中文模式的规则
中文模式里：

+ 所有界面文字是中文
+ 所有操作项后面都标注英文命令
+ 玩家输入时，**统一要求英文命令**
+ 帮助页反复提醒：输入请使用英文命令

例如：

可用操作：  
状态 (status)  
模块 (modules)  
目录 (ls)  
打开 (open)  
扫描 (scan)  
恢复 (restore)  
隔离 (isolate)  
通讯 (relay)  
记录 (record)  
历史 (history)  
身份 (whoami)  
清除 (purge)  
帮助 (help)

这样用户不会懵，而且终端感也保住了。

---

### 3. 英文模式的规则
英文模式就不显示中文：

Available commands:  
status  
modules  
ls  
open  
scan  
restore  
isolate  
relay  
record  
history  
whoami  
purge  
help

---

### 4. 输入为什么必须统一英文
因为如果允许：

+ 中文命令
+ 英文命令
+ 拼音命令
+ 中英混输

很快 parser 会变脏，后面还会影响：

+ 自动补全
+ 命令历史
+ 异常伪命令
+ 英文模式一致性

统一英文输入最稳。

中文模式里只做“显示翻译”和“命令提示映射”。

## 五、操作系统设计：命令要少而硬
不要一开始做 30 个命令。  
第一版控制在 **10~12 个核心命令** 最好。

---

### 1. 全局命令表
| 英文命令 | 中文显示 | 作用 |
| --- | --- | --- |
| `help` | 帮助 | 显示命令帮助 |
| `status` | 状态 | 查看总状态 |
| `modules` | 模块 | 查看模块状态 |
| `ls` | 目录 | 查看目录或文件列表 |
| `open` | 打开 | 打开文件/日志 |
| `scan` | 扫描 | 扫描模块 |
| `restore` | 恢复 | 恢复模块 |
| `isolate` | 隔离 | 切断模块 |
| `relay` | 通讯 | 查看远程消息 |
| `record` | 记录 | 保存快照 |
| `history` | 历史 | 查看命令历史 |
| `whoami` | 身份 | 查看操作者状态 |
| `purge` | 清除 | 执行危险删除操作 |


其实第一版做 12 个以内就够了。  
`purge` 可以后期再上。

如果只有“模块操作”，会太机械。

│   ├─ shift_06.log

│   ├─ handbook.txt  
│   └─ chamber0_note.redacted  
├─ /audio  
│   ├─ cache_01.wav  
│   ├─ residual_trace.txt  
│   └─ voiceprint_report.txt

├─ /auth

“祂是初，祂是终。”

这样氛围有了，实现还轻。

## 七、五个模块要有清晰的“收益 / 风险 / 剧情作用”
这个是整个游戏的骨架。

---

### 1. 音频 `audio`
#### 玩家收益
+ 听到前任录音
+ 听到低频异常
+ 解锁第一条关键颂词

#### 风险
+ 祂获得“声音”
+ 系统开始出现人声提示
+ 通讯对象措辞变异更快

#### 适合出现的句子
> 祂是初，祂是终。
>

---

### 2. 档案 `archive`
#### 玩家收益
+ 读日志
+ 读官方手册
+ 读事故摘要
+ 读到彼此矛盾的文本

#### 风险
+ 文档开始篡改
+ 旧日志会提前写出你当前行为
+ 前任可能就是你自己

#### 适合出现的句子
> 祂是初，祂也是归处。
>

---

### 3. 监控 `camera`
#### 玩家收益
+ 查看收容室文字画面
+ 发现异常位置
+ 看见“空无一物”的 chamber

#### 风险
+ 祂获得“注视”
+ 快照里开始出现不该出现的人影
+ 最后可能出现“你在收容室内”的画面

#### 监控显示推荐做法
不要真图片，直接做“文字快照”。

比如：

> open /camera/frame_B.txt  

快照编号：B  
时间：03:39  

画面描述：  
收容室中央空无一物。  
北侧墙面出现一片不规则阴影。  
阴影边缘近似人形。  
与上一帧相比，其位置更接近观察口。

这特别适合终端。

---

### 4. 权限 `auth`
#### 玩家收益
+ 获取高级协议执行权
+ 查看加密文件
+ 解锁终局操作

#### 风险
+ 系统开始绑定你的身份
+ 你的名字被写进流程
+ `whoami` 开始变化

#### 适合出现的句子
> 祂是初，祂是终，而你是门。
>

---

### 5. 门禁 `lock`
#### 玩家收益
+ 能执行最终收容
+ 能切断门限
+ 能决定“关”还是“断”

#### 风险
+ 系统重新定义“门”
+ 如果你已绑定，门禁操作对象可能不再是房门，而是你

## 八、时间系统：用“动作消耗时间”，不用真实计时
这个一定要做成**伪实时 / 回合式时间**。

因为真实计时会伤害阅读体验。  
你的游戏文本重，真实计时会把玩家逼烦。

---

### 推荐时间消耗规则
| 动作 | 耗时 |
| --- | --- |
| `status` | 0 分钟 |
| `modules` | 0 分钟 |
| `ls` | 0 分钟 |
| `open` | 1 分钟 |
| `scan` | 1 分钟 |
| `relay` | 1 分钟 |
| `record` | 1 分钟 |
| `history` | 0 分钟 |
| `whoami` | 0 分钟 |
| `restore` | 5 分钟 |
| `isolate` | 4 分钟 |
| `purge` | 6 分钟 |


这样玩家会自然产生判断：

+ 我要不要再看一份日志？
+ 我要不要先恢复这个模块？
+ 我要不要现在就去拿权限？
+ 我要不要省时间直接进终局？

这就是紧张感。

---

## 九、剧情推进不能全靠固定段落，要靠“触发条件”
这是你体验感强弱的关键。

建议把剧情事件分成两类：

### 1. 时间触发
到某个时间点自动发生

例如：

+ 03:25 远程第一次催促
+ 03:33 官方手册文本发生第一次变化
+ 03:41 历史命令出现伪命令
+ 03:52 同步预热开始
+ 03:58 命令补全开始失控

### 2. 行为触发
玩家做了某个动作才发生

例如：

+ 恢复音频后第一次出现低语
+ 恢复档案后旧日志提到“归处”
+ 恢复权限后 `whoami` 变化
+ 三次查看同一条异常后，终端弹出那句“三次”
+ 试图否认 / 重开 / 退出后，终端弹出“不要再假装这是第一次看见”

这样剧情就不是死板播报，而是和玩家动作绑在一起。

## 十、你那几句，做成一条“行为绑定链”
这是整部游戏里最有记忆点的文本线索。

不要一次性全给。  
应该做成：

---

### 第一句：恢复音频后
来源：音频残响

> 祂是初，祂是终。
>

玩家只会觉得诡异，还不确定。

---

### 第二句：恢复档案后
来源：早于今夜的旧日志

> 祂是初，祂也是归处。
>

玩家开始怀疑循环和回归。

---

### 第三句：恢复权限后
来源：身份状态页 / 系统回显

> 祂是初，祂是终，而你是门。
>

这里第一次把玩家拖进去。

---

### 第四句：三次重复行为后
触发条件：

+ 三次查看同一条日志
+ 或三次试图否认同一个异常
+ 或三次撤销类似操作

终端弹出：

> 祂是初，祂是终。你已经念过这句话三次了。
>

这句的力点就在“念过”。

---

### 第五句：试图逃避或重置后
触发条件：

+ 玩家尝试退出
+ 尝试当作没看见
+ 尝试重置终端
+ 尝试删除相关记录

终端回应：

> 祂是初，祂是终。不要再假装这是第一次看见。
>

这句最适合晚一点用，杀伤力最大。

## 十一、远程通讯对象要怎么设计才有戏
这个角色非常重要。  
它不是单纯的剧情广播，而是“诱导器”。

---

### 远程对象的表面身份
上一班留下来的远程支援员。

他说话一开始要很正常：

[03:18][远程支援]  
先别碰门禁，先看系统摘要。  

[03:22][远程支援]  
你是新接手的吗？那更要快一点。

---

### 中段开始不对劲
他开始：

+ 知道你没说出口的事
+ 劝你去做高风险恢复
+ 否定日志内容

比如：

[03:31][远程支援]  
不要浪费时间看旧日志。  
那不是写给你现在看的。  

[03:36][远程支援]  
先恢复权限。  
你迟早都得把名字写进去。

---

### 后段彻底暴露
他开始不再像人类支援，而像“某个东西学会了支援员口气”：

[03:49][远程支援]  
你已经走得比前两次更远了。  

[03:55][远程支援]  
不必害怕。  
成为门并不疼。

这里会非常带感。

---

## 十二、三条主要玩法路线
这能保证玩家真的在“决定怎么探索”。

---

### 路线一：保守收容
#### 思路
+ 尽量少恢复模块
+ 尽快执行最低限度协议
+ 相信部分官方系统，不深挖

#### 体验
紧张、压抑、不完全知道真相

#### 结局方向
你可能活下来，但你无法确认自己是否真的阻止了扩散。

结局名：  
**《未完成的夜班》**

---

### 路线二：深度调查
#### 思路
+ 多查日志、多录快照、多对比篡改
+ 怀疑远程
+ 恢复档案、监控、权限

#### 体验
真相浓度高，但绑定风险极高

#### 结局方向
你知道了一切，也完成了自身“门化”。

结局名：  
**《门》**

---

### 路线三：反向利用
#### 思路
+ 假装配合系统
+ 故意顺着诱导恢复部分模块
+ 在终局用清除 / 隔离做反杀

#### 体验
最有操作感，风险最大

#### 结局方向
你阻止了同步，但自己被从系统记录里清除。

结局名：  
**《空白记录》**

---

## 十三、双语显示实现上怎么设计最干净
这里你后面真写代码时会很有用。

不要把中文和英文写死在逻辑里。  
你应该做一个**语言资源表**。

---

### 推荐结构
#### `commands.py`
只定义英文命令：

Python

COMMANDS = {  
"status": {...},  
"modules": {...},  
"ls": {...},  
"open": {...},  
"scan": {...},  
}

#### `i18n_zh.py`
定义中文显示文本：

Python

TEXT_ZH = {  
"title": "零号收容室事故终端",  
"objective": "在 04:00 前阻止异常扩散。",  
"cmd.status": "状态",  
"cmd.modules": "模块",  
"cmd.ls": "目录",  
}

#### `i18n_en.py`
定义英文显示文本：

Python

TEXT_EN = {  
"title": "CHAMBER-0 INCIDENT CONSOLE",  
"objective": "Prevent anomaly propagation before 04:00.",  
"cmd.status": "status",  
"cmd.modules": "modules",  
"cmd.ls": "ls",  
}

#### 渲染层
中文模式输出：

Python

f"{TEXT_ZH['cmd.status']} (status)"

英文模式输出：

Python

"status"

这样整个架构很干净。

## 十四、建议的开发阶段
这个部分最实用。

## 十五、最适合第一版的 UI 风格
不要花太多时间做花哨 ASCII 美术。  
先做这三点就够强：

+ 顶部状态栏
+ 中间信息区
+ 底部命令提示区

再加一些轻量强化：

+ 中文模式用冷白字 + 少量警告红字
+ 时间推进时短促蜂鸣
+ 高危操作前加一段短暂停顿
+ 某些句子单独居中显示

比如你这句就适合单独打：

祂是初，祂是终，而你是门。

不要夹在普通日志里，气势会弱。

## 十六、我建议你现在就固定下来的版本
你现在最应该定的，不是更多文案，而是这 6 件事：

**一，游戏名**  
《门限协议》

**二，输入规则**  
显示可双语，输入统一英文命令

**三，命令集**  
先做 10~12 个

**四，模块集**  
音频、档案、监控、权限、门禁

**五，时间机制**  
动作消耗时间，不做真实计时

**六，核心真相**  
系统不是在关住祂，而是在筛选谁适合成为门

这 6 个一旦固定，项目就能开始写。
